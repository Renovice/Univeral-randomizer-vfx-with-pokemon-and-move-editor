## 1. [crash] PokemonIconCache static CACHE_BY_HANDLER pins every RomHandler (and its full ROM data) forever - OutOfMemoryError after loading several ROMs

**Where:** `src/com/dabomstew/pkrandom/pokemon/editors/PokemonIconCache.java` (~line 25)

**Trigger:** In one session: load a DS/3DS ROM, open the Pokemon editor, close it, load another ROM, open the editor again - repeat. Heap usage grows by roughly the size of each ROM plus icon caches each cycle until the app crashes with java.lang.OutOfMemoryError (typically while scrolling a sheet that decodes icons).

**Detail:** CACHE_BY_HANDLER is a static ConcurrentHashMap<RomHandler, PokemonIconCache> with no eviction. PokemonIconCache.get(romHandler) (lines 40-45) inserts an entry every time an editor is opened for a ROM, and each PokemonIconCache strongly references the RomHandler plus up to ~800 generated ImageIcons. When the user loads a different ROM in the main GUI, the old RomHandler (holding the entire ROM image - hundreds of MB for NDS/3DS games) can never be garbage-collected because this static map keeps it alive even after the editor frame is disposed.

**Suggested fix:**

```
Minimal correct fix (the GUI only ever has one ROM loaded, so a single-entry cache suffices). In PokemonIconCache.java replace the static map with a single current-handler slot that evicts on handler change:

Replace line 25:
    private static final Map<RomHandler, PokemonIconCache> CACHE_BY_HANDLER = new ConcurrentHashMap<>();
with:
    private static PokemonIconCache current;

Replace get() (lines 40-45) with:
    public static synchronized PokemonIconCache get(RomHandler romHandler) {
        if (romHandler == null) {
            return new PokemonIconCache(null);
        }
        if (current == null || current.romHandler != romHandler) {
            current = new PokemonIconCache(romHandler);
        }
        return current;
    }

This bounds retention to exactly one handler (the active one, which the GUI keeps alive anyway) and automatically drops the previous handler + its icons on the next get() with a new ROM. Alternative if multiple live ROMs must be supported: use Collections.synchronizedMap(new WeakHashMap<>()) AND change the instance field to WeakReference<RomHandler> (plain WeakHashMap alone fails because the value strongly references its key). Note the claimant's other alternative (dispose() from a windowClosed handler) also works since windowClosed fires on dispose(), but requires touching every EditorFrame.
```
